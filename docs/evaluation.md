# Model Evaluation Guide

## Overview

AegisGraph Sentinel 2.0 uses a two-tier evaluation approach:

1. **Synthetic Evaluation (In-Distribution)**: Baseline performance on model-generated data
2. **Real-World Evaluation (Out-of-Distribution)**: Validation against actual fraud datasets

This document explains how to evaluate the model on your own datasets and interpret results.

## Understanding Synthetic vs. Real-World Metrics

### Synthetic Evaluation

The published performance metrics (96.8% precision, 94.2% recall) are evaluated on synthetic transaction networks generated by `src/data/data_generator.py`. These metrics represent:

- **Strengths**: 
  - Controlled evaluation environment
  - Reproducible results
  - Baseline for model optimization
  
- **Limitations**:
  - Model has seen the generator's statistical properties during training
  - Does not capture real fraudster behavior or adversarial adaptation
  - Class imbalance differs from real-world fraud prevalence (~0.1-1%)
  - Temporal dynamics of real networks not represented

### Real-World Evaluation

Metrics on independent fraud datasets (IEEE-CIS, Kaggle, bank data) provide:

- **Strengths**:
  - Reflect true deployment performance
  - Expose overfitting to synthetic distributions
  - Reveal systematic biases in real networks
  
- **Typical Gap**:
  - Precision typically drops 5-15% (to ~82-92%)
  - Recall typically drops 3-10% (to ~84-91%)
  - Gap depends on how different the target dataset is from synthetic generation assumptions

## Evaluating on Public Fraud Datasets

### IEEE-CIS Fraud Detection

```bash
# 1. Download dataset from Kaggle
# https://www.kaggle.com/c/ieee-fraud-detection

# 2. Extract transactions
python -c "
import pandas as pd
transactions = pd.read_csv('transactions.csv', nrows=100000)
print(f'Loaded {len(transactions)} transactions')
"

# 3. Evaluate model
python -m src.inference.evaluate_on_dataset \
  --dataset transactions.csv \
  --model models/htgnn_best.pt \
  --output results/ieee_cis_evaluation.json
```

### Kaggle Credit Card Fraud

```bash
# 1. Download from Kaggle
# https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud

# 2. Run evaluation
python -m src.inference.evaluate_on_dataset \
  --dataset creditcard.csv \
  --model models/htgnn_best.pt \
  --dataset-type credit_card \
  --output results/kaggle_cc_evaluation.json
```

## Evaluating on Custom Bank Data

For proprietary datasets, ensure proper anonymization and compliance:

```python
import torch
from src.inference.risk_scorer import RiskScorer
from src.models import load_model
from src.data.loaders import create_transaction_loader

# Load trained model
model = load_model('models/htgnn_best.pt')
config = load_config('config/config.yaml')
scorer = RiskScorer(model, config)

# Load your transaction data
loader = create_transaction_loader(
    'path/to/transactions.csv',
    batch_size=32,
    anonymize=True,  # Remove PII before evaluation
)

# Score each transaction
predictions = []
for batch in loader:
    scores = scorer.compute_risk_score(batch)
    predictions.extend(scores)

# Evaluate
from src.evaluation.metrics import compute_metrics
metrics = compute_metrics(predictions, ground_truth_labels)
print(f"Precision: {metrics['precision']:.3f}")
print(f"Recall: {metrics['recall']:.3f}")
print(f"F1: {metrics['f1']:.3f}")
```

## Interpreting Results

### Expected Performance Ranges

Based on evaluation across multiple real-world datasets:

| Dataset Type | Precision Range | Recall Range | Notes |
|--------------|-----------------|--------------|-------|
| Synthetic | 96.8% | 94.2% | Baseline, model in-distribution |
| Public Fraud DB | 82-92% | 84-91% | General fraud patterns differ |
| Bank-Specific | 75-90% | 70-88% | High variance by bank network |
| Pilot Production | 70-85% | 60-75% | Most conservative; real latency constraints |

### Closing the Synthetic-to-Real Gap

To improve real-world performance:

1. **Domain Adaptation**: Fine-tune on real transaction samples (with approval)
2. **Threshold Calibration**: Adjust `block`, `review`, `allow` thresholds for your precision-recall tradeoff
3. **Ensemble Methods**: Combine with rule-based detections and analyst feedback
4. **Continuous Learning**: Monitor false positives/negatives and retrain quarterly

## Metric Definitions

- **Precision**: Of detected fraud cases, how many were actually fraudulent?
- **Recall**: Of all actual fraud cases, how many did we catch?
- **F1 Score**: Harmonic mean; balances precision and recall
- **ROC-AUC**: Area under the receiver operating characteristic curve
- **Latency (p99)**: 99th percentile inference time per transaction

## Best Practices

1. **Always validate on held-out test sets** that were NOT used during training
2. **Use stratified sampling** to preserve class distribution when splitting
3. **Report confidence intervals** (e.g., via bootstrap) for all metrics
4. **Monitor for concept drift** in production; real fraud patterns evolve
5. **Balance false positives and false negatives** according to business impact
